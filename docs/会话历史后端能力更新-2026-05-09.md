# 会话历史后端能力更新（2026-05-09）

本文档用于记录本次在后端完成的“会话历史与继续对话”能力，便于后续回溯设计原因、接口语义与目录结构。

本次改动的目标不是替换既有 `thread_id` 机制，而是在其基础上补齐：
1. 完整会话消息持久化
2. 会话列表/详情查询接口
3. 服务端创建会话与恢复旧会话能力

## 1. 本次完成项总览

已完成：
1. 新增 `ConversationStore`，负责完整会话摘要与消息历史持久化。
2. `ChatService` 支持在未传 `thread_id` 时由服务端自动创建新会话。
3. `ChatService` 每轮对话后同时更新：
   - `context.json`
   - `conversation.json`
   - `messages.json`
4. 新增会话查询与创建接口：
   - `GET /conversations`
   - `GET /conversations/{thread_id}`
   - `POST /conversations/create`
   - `POST /conversations/{thread_id}/rename`
5. `/chat` 响应补充 `thread_id`，流式 `done` 事件继续返回 `thread_id`。
6. 模型历史恢复优先使用完整会话消息，再回退到轻量上下文与旧前端兼容字段。

关键文件：
1. `harness/context/conversation_store.py`
2. `harness/context/thread_runtime.py`
3. `APP/service/chat_service.py`
4. `APP/api/app.py`
5. `APP/dto/chat.py`
6. `harness/models/types.py`

## 2. 端到端请求链路（当前行为）

### 2.1 会话创建与发送

1. 前端可先调用 `POST /conversations/create` 显式创建会话。
2. 若前端直接调用 `/chat` 或 `/chat/stream` 且未传 `thread_id`，后端会自动生成新的 `thread_id`。
3. `ChatService` 为该 `thread_id` 创建 thread 目录，并确保会话文件存在。
4. agent 运行完成后：
   - `context.json` 写入轻量上下文
   - `conversation.json` 写入会话摘要
   - `messages.json` append 完整 user/assistant 消息

### 2.2 恢复旧会话继续对话

1. 前端通过 `GET /conversations` 获取会话列表。
2. 前端通过 `GET /conversations/{thread_id}` 获取完整历史。
3. 用户继续提问时继续传相同 `thread_id`。
4. `ChatService` 重新定位该 thread 目录并加载：
   - 完整消息历史
   - 轻量上下文
   - 该 thread 的 workspace / uploads / outputs / skills
5. 即使上一轮请求的协程、agent loop、处理线程已经结束，只要 thread 目录仍在，会话仍可恢复。

## 3. 三类持久化文件的职责

当前每个 thread 目录下会存在三类核心 JSON：

### 3.1 `context.json`

职责：模型续聊所需的轻量上下文。

核心字段：
1. `thread_id`
2. `recent_messages`
3. `pinned_skills`
4. `compact_summary`
5. `updated_at`

特点：
1. 会裁剪与压缩。
2. 不保证完整回放。
3. 主要服务于提示词与上下文治理。

### 3.2 `conversation.json`

职责：会话摘要元数据，主要供会话列表展示使用。

核心字段：
1. `thread_id`
2. `title`
3. `created_at`
4. `updated_at`
5. `last_message_preview`
6. `message_count`

特点：
1. 标题首版使用首条用户消息截断生成。
2. `last_message_preview` 优先来自最后一条 assistant 回复。
3. 该文件不存完整正文。

### 3.3 `messages.json`

职责：完整可回放消息历史。

核心字段：
1. `id`
2. `role`
3. `content`
4. `created_at`

当前行为：
1. 仅保存 `user` / `assistant` 消息。
2. 不保存 subagent 流式事件。
3. 不参与技能命中与 compact 逻辑。

## 4. 历史恢复优先级（当前版本）

`ChatService` 当前组装模型 history 的优先级为：
1. 优先读取 `messages.json`，裁成最近窗口作为模型历史。
2. 若完整会话历史不存在，则回退到 `context.json.recent_messages`。
3. 若仍不可用，再回退到旧前端传入的 `messages` 字段。

这样做的原因：
1. 恢复旧 session 时，模型能看到真实的完整对话历史。
2. `context.json` 仍然保留轻量 compact 优势。
3. 旧前端协议仍可兼容。

## 5. Thread 目录结构（当前版本）

当前 thread 目录结构为：

```text
.sandbox/threads/<normalized_thread_id>/
  context.json
  conversation.json
  messages.json
  workspace/
  uploads/
  outputs/
  skills/
```

说明：
1. 目前三个 JSON 仍直接位于 thread 根目录。
2. 这是当前可接受实现，不影响功能正确性。
3. 后续如需进一步整理，可迁移到单独的 `state/` 或 `metadata/` 子目录。

目录语义约定：
1. `workspace/`：agent 当前 thread 的工作区文件
2. `uploads/`：用户上传原件
3. `outputs/`：生成结果与导出产物
4. `skills/`：会话级技能副本
5. 根目录 JSON：当前版本的系统状态与会话元数据

## 6. 接口语义（当前版本）

### 6.1 既有聊天接口

#### `POST /chat`

请求：
1. `message`
2. `model`
3. `thread_id`（可选）
4. `messages`（兼容旧前端，可选）

响应：
1. `text`
2. `trace`
3. `thread_id`

行为：
1. 未传 `thread_id` 时自动创建新会话。
2. 传已存在 `thread_id` 时继续该会话。

#### `POST /chat/stream`

行为与 `/chat` 一致，`done` 事件中额外回传：
1. `text`
2. `trace`
3. `thread_id`

### 6.2 新增会话接口

#### `GET /conversations`

返回：
1. 会话摘要列表
2. 默认按 `updated_at desc` 排序

#### `GET /conversations/{thread_id}`

返回：
1. 单个会话详情
2. 包含完整 `messages`

#### `POST /conversations/create`

返回：
1. 新会话摘要
2. 由服务端生成 `thread_id`

#### `POST /conversations/{thread_id}/rename`

请求：
1. `title`

返回：
1. 更新后的会话摘要

## 8. 与 2026-04-19 文档的差异

相比 [skills与上下文治理更新-2026-04-19.md](./skills与上下文治理更新-2026-04-19.md)，当前系统有两点关键变化：

1. `thread_id` 不再必须由前端页面生成。
当前后端可以在未传 `thread_id` 时自动创建新会话。

2. 线程上下文与完整会话历史已明确拆层。
`context.json` 负责轻量上下文治理，`conversation.json + messages.json` 负责前端可见会话历史。

这意味着“线程级记忆”和“会话可回放”已经不再是同一个 JSON 文件承担的能力。
